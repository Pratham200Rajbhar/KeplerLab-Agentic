Distributed Systems & Cloud Computing

From RPC to Web Services to Cloud Infrastructure

  • Distributed systems: independent computers appearing as a single coherent system
  • Middleware: software layer masking heterogeneity and enabling distributed programming
  • Cloud computing: utility-based delivery of virtualized computing resources over the Internet

Understanding distributed computing foundations is essential to grasp how cloud computing evolved and how modern web services operate.

Remote Procedure Call (RPC) Architecture

RPC enables a process to call a subroutine in a different address space, typically on another machine, using stubs and network communication.

  • Client stub marshals parameters and sends message over network
  • Server stub unmarshals parameters and invokes target procedure
  • Network routines handle message transmission between stubs
  • Stubs act as proxies, making remote calls appear local to developers
Image: Side-by-side diagram showing client and server machines with processes, stubs, and network routines; arrows illustrate data flow from client call through stub, network, server stub, to server procedure

RPC Execution Steps

RPC involves 10 distinct steps from client invocation to result return, ensuring transparent remote execution despite network complexity.

  • 1. Client calls stub with local procedure semantics
  • 2. Stub builds message and sends via network routines
  • 3. Message transmitted across network
  • 4. Server OS delivers message to server stub
  • 5. Server stub unmarshals parameters and calls server function
  • 6. Server function executes and returns
  • 7. Server stub marshals return value and sends response
  • 8. Client stub receives and unmarshals response
  • 9. Client stub returns result to client code
  • 10. Client resumes execution with result

Web Services Fundamentals

Web services are self-contained, platform-independent, and loosely coupled software components accessible over the Internet.

  • Machine-oriented (unlike websites for humans); designed for application-to-application communication
  • Use standard protocols: HTTP, XML, SOAP, REST, WSDL, UDDI
  • Solve interoperability, firewall traversal, and complexity issues of earlier distributed systems
Problem SolvedSolution
InteroperabilityStandard XML-based messaging across platforms
Firewall TraversalUse HTTP port 80, widely allowed through firewalls
ComplexityDeveloper-friendly with standardized interfaces

Web Services Architecture Stack

Web services rely on XML-based technologies for messaging, description, and discovery—forming a layered architecture for service interaction.

  • XML Messaging: SOAP messages as XML streams over HTTP
  • WSDL: XML-based interface definition for service requester-provider interaction
  • UDDI: Registry for publishing and discovering web services
Image: Three-layer vertical stack: bottom layer XML (messaging), middle layer WSDL (description), top layer UDDI (registry); arrows show flow from discovery to invocation to response

Web Services Lifecycle

A web service operation follows three core phases: publish, find, and bind—enabling dynamic service discovery and integration.

  • Publish: Service provider registers WSDL in UDDI registry
  • Find: Client queries registry to locate suitable service
  • Bind: Client uses WSDL to invoke service and exchange messages

IaaS Characteristics & Benefits

Infrastructure as a Service provides virtualized computing resources over the internet with dynamic scaling, utility pricing, and self-service provisioning.

  • Dynamic Scaling: Resources automatically scale based on demand
  • Rental Model: Pay-per-use pricing without owning physical hardware
  • Self-Service Provisioning: Users provision resources via web interfaces
  • No Hardware Investment: Provider manages physical infrastructure
  • Location Independence: Accessible from anywhere with internet
  • No Single Point of Failure: Redundancy across data centers

Virtualization in Cloud Infrastructure

Virtualization decouples OS from hardware, enabling efficient resource pooling, migration, and isolation—core to cloud elasticity.

  • Server Virtualization: Multiple VMs run on one physical server via hypervisor
  • Hypervisor Types: Type 1 (bare-metal, e.g., ESXi) vs Type 2 (hosted, e.g., VirtualBox)
  • Full Virtualization: Unmodified guest OS (e.g., VMware), uses binary translation
  • Para-Virtualization: Modified guest OS for performance (e.g., Xen)
  • Live Migration: Move running VMs between hosts with minimal downtime
