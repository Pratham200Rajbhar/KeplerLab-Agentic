XML vs HTML: Foundational Data Technologies

Design, Purpose, and Structural Differences

  • XML: Extensible, content-oriented, transport/storage of data; user-defined tags and attributes
  • HTML: Fixed-tag, presentation-oriented, display of data; pre-defined tags for visual formatting
  • XML enables validation via schemas; HTML offers no built-in data validation
  • XML supports namespaces and metadata; HTML focuses on rendering semantics (e.g., <i>, <br>, <font>)

XML is designed for data transport and storage with extensibility; HTML is rigid and designed for human-facing presentation.

XML Document Structure and Validation

Syntax, Namespaces, and Schema Integration

  • XML documents begin with a declaration (e.g., <?xml version="1.0" encoding="UTF-8"?>) and may include comments
  • Namespaces (xmlns) and schemaLocation enable modular schema binding and avoid tag-name collisions
  • Elements can carry attributes (e.g., year="2000") and nested child elements (e.g., <author>, <publisher>)
  • Validation requires external schema (e.g., DocBook XSD) — XML itself provides no native validation

Web Services Communication: SOAP Protocol

Envelope, Body, and Structured Data Exchange

  • SOAP uses XML-based envelope structure with namespace declarations for envelope, encoding, and schema
  • The <SOAP-ENV:Body> contains the actual request/response payload (e.g., <calculateCarPayment>)
  • Parameters are typed using XSD types (e.g., xsi:type="xsd:string") for interoperability
  • EncodingStyle attribute defines serialization rules for data within the body

Distributed System Interoperability Requirements

Cross-Platform and Cross-Environment Compatibility

  • Distributed systems must operate across heterogeneous environments: OS, hardware, languages, and security models
  • Openness requires independence from underlying heterogeneity — hardware, platforms, and programming languages
  • Standards like CORBA, DCOM, and Web Services enable abstraction and communication across such boundaries
  • Information representation must be standardized (e.g., XML, JSON) to ensure semantic consistency

Full Virtualization: Binary Translation & Direct Execution

Guest OS Transparency and Performance Trade-offs

  • Full virtualization simulates complete hardware abstraction — guest OS runs unmodified and unaware of virtualization
  • Uses binary translation for privileged/sensitive instructions; direct execution for non-sensitive user-level code
  • Significant overhead due to translation layer; no hardware/OS assistance required
  • Commercial: VMware ESX, Microsoft Virtual Server; Open-source: Xen HVM, KVM

Virtualization Architecture Layers

Hardware, Hypervisor, and Virtual Machine Abstraction

  • Hardware layer: Physical components — CPU, memory, NICs, disks
  • Hypervisor layer: Thin software layer managing resource allocation and providing dedicated virtual views
  • Virtual machine layer: OS instances appearing as physical machines; portable due to hardware abstraction
  • VMs share physical resources but maintain isolation and portability across hosts
