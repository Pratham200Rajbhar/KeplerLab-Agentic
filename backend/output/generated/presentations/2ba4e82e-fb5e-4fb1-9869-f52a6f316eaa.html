Web Services and Cloud Computing

Foundations, Architectures, and Relationships

Web Services, SOA, and Cloud Computing form an interrelated ecosystem enabling distributed computing over the Internet. This presentation clarifies their definitions, distinctions, and technical foundations.

Web Services, SOA, and Cloud Computing: Definitions and Relationships

  • Cloud Computing is a broad term for any Web service software, hardware, and applications.
  • SOA (Service-Oriented Architecture) focuses mainly on software services and their interactions.
  • Web Services encapsulate Cloud Computing connections because Cloud Computing uses Web Services for communication.
  • Cloud Computing and SOA are independent approaches—neither requires the other to function.

Web Services can be used outside Cloud Computing, such as in on-premises SOA. Conversely, SOA can exist without Web Services, using other communication mechanisms.

Remote Procedure Call (RPC) Architecture

  • In a local procedure call, the calling and called processes share the same address space.
  • In a remote procedure call, processes may run on different machines and communicate over a network.
  • Stubs (client and server) act as proxies: marshaling parameters and return values across the network.
  • RPC involves 10 steps: client call → stub call → message build → network send → message receive → stub unpack → server call → return → marshal result → client return.
StepAction
1Client calls stub
2Stub builds message
3Message sent over network
4Server stub receives and unpacks
5Server function invoked
6Result marshaled and sent back

Distributed Object Technologies: CORBA vs. RMI

  • CORBA (Common Object Request Broker Architecture) supports cross-language object communication using IDL (Interface Definition Language).
  • RMI (Java Remote Method Invocation) is Java-specific and simpler, supporting only Java-to-Java communication.
  • CORBA is complex and historically dominant; Microsoft transitioned from CORBA to COM to .NET.
  • Both use stubs and skeletons: RMI serializes parameters on the client stub and deserializes on the server skeleton.

Web Services Architecture and Key Technologies

  • Web Services are self-contained, platform-independent, and can be described, published, located, and invoked over the Internet.
  • Web Services solve three key problems: interoperability, firewall traversal, and complexity reduction.
  • XML is the foundation: all messaging, description, and registry use XML for platform independence.
  • Core technologies: SOAP (protocol), WSDL (interface definition), UDDI (service registry).

Web Services differ from websites: websites serve humans; Web Services serve machines (applications).

SOAP vs. REST Web Services

  • SOAP is a protocol; REST is an architectural style.
  • SOAP uses XML exclusively; REST supports XML, JSON, HTML, plain text.
  • SOAP defines its own security; REST inherits security from underlying transport (e.g., HTTPS).
  • REST requires less bandwidth and is more preferred for web APIs.
  • SOAP uses service interfaces; REST uses URIs to expose business logic.

Web Services Lifecycle and WSDL Structure

  • Web Services follow a find-publish-interact lifecycle: publish service, discover via registry, interact via SOAP.
  • WSDL defines the service interface: data types, messages, operations, bindings, port, and service endpoint.
  • WSDL is XML-based and acts as an IDL for Web Services, enabling client stub generation.
  • A WSDL document includes: <types>, <message>, <portType>, <binding>, <port>, and <service>.
WSDL ElementPurpose
<types>Data type definitions
<message>Abstract data definitions (request/response)
<portType>Abstract operations (input/output)
<binding>Protocol and format (e.g., SOAP/HTTP)
<service>Endpoint address and port aggregation

UDDI Registry and Service Discovery

  • UDDI (Universal Description, Discovery, and Integration) serves as a business and service registry for Web Services.
  • Businesses populate UDDI with service descriptions; clients query it to discover services.
  • UDDI enables find/lookup of services, retrieval of WSDL, and invocation via SOAP.
  • UDDI supports public, private, and hybrid registry deployments.

UDDI is part of the Web Services stack: after publishing a service, clients use UDDI to locate and bind to it.

Cloud Computing: Core Concepts and Characteristics

  • Cloud Computing is Internet-based computing that provides shared resources and data to devices on demand.
  • It hides infrastructure complexity via simple APIs or graphical interfaces.
  • Key characteristics: massive scale, resilience, virtualization, service orientation, low cost, and advanced security.
  • Essential characteristics: on-demand self-service, broad network access, resource pooling, rapid elasticity, and measured service.

Cloud Computing decouples services from hardware and OS, enabling migration, elasticity, and reduced operational overhead.

IaaS, PaaS, SaaS: Cloud Service Models

  • IaaS provides virtualized computing resources over the internet (e.g., VMs, storage, load balancers).
  • PaaS delivers hardware and software tools over the internet—typically for application development.
  • SaaS delivers software applications over the internet on a subscription basis (e.g., Gmail, Salesforce).
  • IaaS is the foundational layer; PaaS builds on IaaS; SaaS is the topmost, end-user layer.

Virtualization and Hypervisors

  • Virtualization separates the OS from hardware, enabling multiple VMs on one physical server.
  • Hypervisors manage VMs: Type 1 (bare-metal, e.g., VMware ESXi, Xen) and Type 2 (hosted, e.g., VMware Workstation).
  • Full virtualization simulates hardware (no guest OS modification); para-virtualization requires guest OS changes for performance.
  • Hardware-assisted virtualization (Intel VT-x, AMD-V) improves performance by offloading privileged instruction handling to the CPU.

Distributed Systems and Middleware

  • A distributed system is a collection of independent computers that appears as a single coherent system to users.
  • Key properties: transparency (access, location, failure), openness, scalability, concurrency, heterogeneity, and fault tolerance.
  • Middleware is a software layer between OS and applications that masks heterogeneity and provides programming abstractions.
  • Examples: RPC, RMI, CORBA, DCOM, and database connectivity APIs (ODBC, JDBC).

Middleware enables transparent communication across heterogeneous platforms, forming the backbone of distributed object systems and Web Services.

Grid vs. Cloud Computing

  • Grid Computing: heterogeneous, geographically dispersed, resource sharing across administrative domains (e.g., scientific research).
  • Cloud Computing: homogeneous, single-domain, utility-based provisioning with SLAs (e.g., AWS, Azure).
  • Grid emphasizes solving large-scale problems; Cloud emphasizes scalable, standard environments for development and deployment.
  • Grid uses utility computing for resource negotiation; Cloud uses simple user→provider models with pay-per-use.

Advantages and Disadvantages of Cloud Computing

  • Advantages: lower costs, reduced software licensing, improved performance, unlimited storage, universal access, automatic updates, and enhanced reliability.
  • Disadvantages: requires constant internet, may be slow, stored data may be insecure, potential vendor lock-in, and limited feature sets compared to desktop apps.
  • IaaS advantages: no hardware investment, utility-style costing, scalability, location independence, and no single point of failure.
  • IaaS disadvantages: hardware/software constraints, legal issues with data location, security gaps, and latency for high-speed interactions.
