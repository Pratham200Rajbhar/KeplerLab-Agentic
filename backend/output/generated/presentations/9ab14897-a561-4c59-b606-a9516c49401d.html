Distributed Systems: Core Principles

Independence, Interoperability, and Transparency

  • Independence from heterogeneity: hardware, platforms, and languages
  • Interoperability across open systems via standardized interfaces
  • Transparency as a key desired property

Distributed systems must abstract environmental diversity while enabling seamless interaction across heterogeneous components.

Transparency in Distributed Systems

Eight Key Forms of Transparency

Transparency TypeDescription
AccessHide differences in data representation and resource access
LocationHide where a resource is located
MigrationHide that a resource may move to another location
ReplicationHide that a resource is replicated
ConcurrencyHide that a resource may be shared by several competitive users
FailureHide the failure and recovery of a resource

Transparency ensures users and applications interact with resources without needing to know underlying implementation details.

Desired Properties of Distributed Systems

Core Objectives for System Design

  • Transparency: Uniform access regardless of location or implementation
  • Openness: Conformance to well-defined interfaces and portability
  • Scalability: Ability to grow in size and complexity without degradation
  • Enhanced Availability: Fault tolerance and redundancy support
  • Concurrency: Support for multiple users accessing shared resources
  • Heterogeneity: Independence from underlying platform differences

These properties collectively enable robust, scalable, and maintainable distributed systems that serve users and resources efficiently.

Web Services Architecture

Standards-Based Interaction Framework

  • WSDL defines the interface for service interaction (external IDL)
  • UDDI serves as a registry for discovering and publishing services
  • SOAP enables structured message exchange with built-in security
  • XML provides extensible, content-oriented data transport

Web services rely on standardized, XML-based protocols to enable application-to-application communication across heterogeneous environments.

Image: A flow diagram: Web Service Requester → UDDI Registry (search icon) → Web Service Provider (server icon) → SOAP Request/Response (envelope icon). Include WSDL document icon attached to the provider, and XML tags (e.g., <book>, <author>) floating near the SOAP envelope

WSDL Components

What WSDL Describes

  • Data types used by the service
  • Parameters required and returned by operations
  • Groupings of functionality (portTypes)
  • Protocol used to access the service (e.g., SOAP over HTTP)
  • Service endpoint location (address)

WSDL acts as a contract that enables clients to understand how to interact with a service without prior knowledge of its implementation.

XML vs. HTML

Purpose, Structure, and Use Cases

FeatureXMLHTML
TagsExtensible, user-definedFixed, pre-defined
OrientationContent-orientedPresentation-oriented
ValidationSupports schema validation (XSD)No built-in validation
Primary UseTransport and store dataDisplay data to humans

XML is designed for data transport and validation; HTML is designed for visual presentation and rendering.

Self-Service in IaaS

Business Value and Operational Efficiency

  • Self-service is a core imperative in IaaS—enables direct user control
  • Analogous to banking ATMs: reduces need for manual intervention
  • Customers make routine requests conforming to predefined business rules
  • Enables automation, reduces operational overhead, and improves scalability

Self-service interfaces empower users to provision and manage resources on demand, mirroring the efficiency of automated service models.

Web Services Characteristics

Key Architectural Traits

  • Loose Coupling: Minimizes dependencies between services
  • Standard-Based: Ensures interoperability via open standards
  • Flexibility & Reusability: Services can be composed and reused
  • Scalability & Programmatically Accessible: Designed for growth and automation
  • Application-to-Application Communication: Enables enterprise integration

These traits support resilient, maintainable, and evolvable service-oriented architectures.
